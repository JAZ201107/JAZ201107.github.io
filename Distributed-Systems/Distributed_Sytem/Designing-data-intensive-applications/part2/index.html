Designing Data Intensive Application Part2

Posted on 2022-04-07 Edited on 2022-04-08 In Distributed Systems Views:
Symbols count in article: 5.9k Reading time ≈ 5 mins.

第一部分讨论了单台机器存储系统的主要技术。第二部分讨论的是多台机器提供数据存储（storage）和检索服务（retrieval of data）

[toc]

前言

有以下的几个目的：

  1. 扩展性（Scalability）： 数据量超出单台机器的处理上线
  2. 容错（Fault Tolerant）和高可用性（high availability）：通过多台机器提供冗余。
  3. 低延迟性（latency）：就近提供数据，避免跨越半个地球才能到达目标
  4. 安全性（MIT 6.824）：通过isolation，来获取security

Scaling to Higher Load

有共享的结构：（vertical scaling or scaling up）

  • shared memory architecture
  • shared disk architecture

当需要更强的处理能力时，首先是：

购买更强大的机器， 共享内存结构（shared memory architecture），但是这个方法的问题是：

  1. 成本高：CPU增加一倍，RAM增加一倍 等， 所需的成本不止增加一倍
  2. 性能不好：由于性能瓶颈的原因，有两倍硬件指标，不一定能处理两倍负载
  3. 有限的容错能力：局限于某个特定的地理位置，无法提供异地容错能力

另一种方法 共享磁盘架构（shared disk architecture）：

有多台服务器，独立的CPU和RAM，共享的磁盘阵列（array of disks），通过高速网络连接（NAS或SAN）。存在的问题：

资源竞争和锁的开销，限制其扩展能力。

什么是 NAS（ Network Attached Storage）：

连接在网络上，具备资料存储功能的装置，因此也称为“网络存储器”。它是一种专用数据[存储服务器]。它以数据为中心，将存储设备与服务器彻底分离，集中管理数据，从而释放带宽、提高性能、降低总拥有成本、保护投资。其成本远远低于使用服务器存储，而效率却远远高于后者。

支持多种协议（如NFS、CIFS、FTP、HTTP等）

什么是 SAN（Storage Area Network）： 企业最常用的存储网络架构，要求高吞吐量和低延迟的业务关键型应用往往采用这类架构运行。

两者区别：

![](Screenshot 2022-04-08 at 00.47.38.png)

无共享的结构（horizontal scaling or scaling out）

运行数据库软件的机器或者虚拟机叫做节点（node）， 每个node有自己的 CPU，RAM 和 disk。node 之间的合作通过传统网络（以太网），在软件层面实现。

优点：

  • 不需要专门的硬件：有较高的性价比
  • 跨地区分布：减少延迟
  • 高容错能力：数据中心发生灾难时，仍可以工作

缺点：

  • 更多复杂性
  • 可能会限制实际可用的数据模型

复制（replication） 与 分区（partitioning）

数据分布多nodes有两种常见的办法：

  • replication： 多个nodes保持相同的数据副本
  • partitioning： 将一个大的数据库拆分为小的dataset叫做partitions。因此不同的partitions可以分不在不同的nodes上（sharding）

这两种不同的机制，通常和在一起用（eg： MapReduce）

复制（Replication）

有三种常用的replication的方法：

  • single-leader（主从复制）
  • multi-leader（多主节点复制）
  • leaderless（无主节点复制）

主节点与从节点（Leader and Follwer）

每个保存数据库完整数据集的节点称为 副本。

问题：如何保持副本的一致性？

最常见的方法： 主从复制。

![Screenshot 2022-04-08 at 10.20.47](/Users/zhangyuyang/Desktop/blog/source/_posts/Distributed_Sytem/Designing-data-intensive-applications/part2/Screenshot 2022-04-08 at 10.20.47.png)

  1. 指定一个副本为leader（或叫做master or primary），当用户想写入数据库时，必须将请求先发送给leader，leader首先将数据写入本地存储
  2. 其他replica称作follower（或 salve，secondaries）。当leader把数据写入local storage之后，通过 replication log 或者 change stream的形式，将数据的变化发送给 follower。follwer获得log之后，将其更新到本地，且严格保持与leader 写入的顺序相同
  3. client可以从，leader或者 follower读取数据，但只能通过leader写数据

同步（Synchronous）与异步（Asynchronous）复制

![Screenshot 2022-04-08 at 10.29.19](/Users/zhangyuyang/Desktop/blog/source/_posts/Distributed_Sytem/Designing-data-intensive-applications/part2/Screenshot 2022-04-08 at 10.29.19.png)

follower1 的复制是同步的： leader需要等待follower1确认变更了数据，才会向用户报告，并且将最新的写入对其他客户端可见

follower2 的复制是异步的：leader发送给follwer2之后，立即回应给client，无需等到follwer2的回应

同步：

  • 优点：一旦leader返回给client， 就说明follower一定完成了与leader的更新同步
  • 缺点：一旦follower没有response（follower崩溃或网络阻塞），写入就不能视为成功。leader会阻塞其后所有的写操作，直到replica完成

在实际开发中，如果数据库采用了 同步复制，意味着： 某一个follower是同步的，其他的异步的，如果这个同步的follower不可用或者性能下降，那么将另一个异步的follower提升为同步模式。这种配置叫做半同步（semi-synchronous）

异步：

  • 优点：leader可以持续相应写的请求，即使follower 数据滞后，吞吐性能更好
  • 缺点：如果leader发生失败或不可恢复，那么尚未复制的写请求将丢失，这意味着，即使向client确认了写的操作，却无法保证数据的持久化（durable）

在follower数量巨大，或者follower分布距离广泛的情况下，异步复制被广泛的使用。

研究人员尝试通过别的方法来提高复制性能和可用性：

eg：链式复制（chain replication），是一种同步复制的变体。

新增follwer（Setting up new follwers）

如何确定新增follwer与leader的数据一致性？

  1. 简单的将文件从一个节点复制到另一节点？

存在问题： client仍在写入数据，数据始终在变化，这种方法会导致不同节点上呈现不同时间点的数据。

  1. 在服务不中断的前提下完成添加新的follower。
    1. 在某个时间点上，对leader的数据库产生一个consistent snapshot。这样避免了长时间锁定数据库
    2. 将snapshot拷贝到新的follower
    3. 新的follower连接到leader，并请求snapshot之后，所有的数据更改日志。这需要snapshot与系统replication log的某个确定位置相关联。
    4. follwer处理data changes的日志，改过程称作caught up。 重复步骤 1～4.

处理节点失效（Handing Node Outages）

目标：尽管个别节点出现中断，保持系统总体运行，并且尽可能减少节点中断带来的影响。

Follwer 失效：catch-up recovery。

follwer在本地磁盘保存 log of data changes， 当follower在崩溃后恢复，可以通过log快速的知道，最后一个事务（transaction）。然后连接到leader，并且请求在最后一个事务之后的所有data change。当其catch up之后，就可以继续接受，来自主节点的数据流的变化。

Leader 失效： Failover

什么是Failover：

  • leader失效，选择将某个从节点提升为主节点；
  • clients需要重新配置，将请求发送给新的leader；
  • 其他follwer需要接受新的leader的data change；

这一过程称作 failover。

failover可以手动（通知管理员发生Leader failure，采取必要的步骤来创建新的leader），或者自动。自动的步骤如下：

  1. 确定leader失效：大多数系统系统运用一种超时机制（通过互相发送hearbeat。如果发现某个node 在一段比较长的时间（timeout）内没有response，则认为该node失效）
  2. 选择新的leader： 通过election process（leader 通过majority of the remaining replicas）。或者通过之前选定的 controller node
  3. 重新配置系统（reconfiguring the system to use the new leader）：当old leader 恢复了，要 确保old leader变成follower

但是上述过程存在很多变数：

  • 如果使用异步的方式，new leader 可能没有接收到原来leader的所有数据。解决办法之一就是丢弃未完成复制的写请求
  • 在数据库之外有其他系统依赖于数据库的内容并在一起协同使用，那么丢弃数据的方案特别危险
  • 在一些特殊情况下，两个nodes 都相信他们是leader，这种情况叫做 split brain。当两个leader都收到数据，但是没办法解决conflicts，就会导致数据丢书或者破坏
  • 什么timeout的时间才是适当的。

node failures，unreliable network，trade-off， replica consistency， durability，availability，和 latency是分布式系统的核心基本问题

复制日志的实现（Implementation of Replication Logs）

leader-based replication的不同实现方法：

  1. 基于语句的复制（Statement-based replication）
  2. 基于预写日志（WAL）传输（Write-ahead log（WAL）shipping）
  3. 基于行的逻辑日志复制（Logical（row-based）log replication）
  4. 基于触发器的复制（Trigger-based replication）

基于语句的复制（Statement-based replication）

leader对于每个写的请求（statement），执行，并且将statement log 发送给followers。（对于relational database，就是增删改）。每个follower分析并执行 statement。

不适用场景：

  • 任何调用非确定性函数（nondeterministic function）比如 NOW( )， RAND( )， 可能会在副本上产生不同的值
  • 如果采用了auto incrementing column，或者需要数据库现有的数据，那么，所有replica必须按照完全相同的顺序执行。 因此，如果有多个同时进行的事务（multiple concurrently executing transactions）， 会有很大的限制
  • 有副作用的statement（如，触发器，用户自定义函数），可能会导致在不同副本上有不同的副作用

基于预写日志（WAL）传输（Write-ahead log（WAL）shipping）

通常每个写操作都是以追加写的方式写入日志中的。因此可以通过完全相同的日志，在另一个node上构造replica：leader 通过 网络将其发送给node

存在问题：log描述的data是非常底层的（low-level），这使得replication 和 storage engineer 紧密耦合（coupled to）。如果storage format从一个版本到另一个版本，那么系统无法支持leader 和 follwer运行不同的版本

基于行的逻辑日志复制（Logical（row-based）log replication）

replication 和 storage engineer 采用不同的 log 格式， 使得replication log 和 storage enginner 剥离（decoupled）。这种方式叫做 逻辑日志（logical log）。

relational database的logical log 通常是指一系列描述数据库行级别的写请求的记录。（records describing writes to database tables at the granularity of a row）：

  • 对于 inserted row， log 包含所有相关列的值
  • 对与 deleted row，log 包含足够信息来唯一标识已删除的行，通常是主键，如果没有主键，则记录所有column的旧值。
  • 对于 updated row，log 包含足够信息来唯一标识更新的行，以及所有列的新值。

如果一条transaction涉及多行的修改，则会产生多个logs记录，并在后面更着一条记录，指出该transaction已经提交（committed）

基于触发器的复制（Trigger-based replication）

某些情况下，需要更高的灵活性，如复制部分数据，或者从一份数据库复制到另一份数据库，那么需要将复制控制权交给应用程序层（Application layer）：

  1. 有一些工具，例如Oracle GoldenGate， 可以读取数据的日志让应用程序获取数据变更。
  2. 另一种是自带的过程： triggers and stored procedure。
